RehanPulse — Developer Activity Command Center
Know exactly what your codebase is doing, in real time, from one elegant dashboard.
RehanPulse aggregates your GitHub commits, pull requests, CI runs, Vercel deployment statuses, and Firebase usage metrics into a single real-time command center. Think macOS Activity Monitor — but for your entire dev lifecycle. It's both innovative (unified observability for solo devs & small teams) and immediately practical (you use it every day).

RehanPulse — Project Blueprint
Project idea — RehanPulse aggregates GitHub, Vercel, and Firebase into a single macOS-inspired command center. It's genuinely useful daily, and every feature naturally exercises one or more of your six engineering pillars.
Tech stack — Next.js 14 App Router + TypeScript (strict) + Tailwind + shadcn/ui for the UI; Firebase Auth + Firestore for the backend; Zustand + React Query for state; Framer Motion for the macOS feel.
Architecture — The most critical decision: SSE over WebSockets, because Vercel's free tier doesn't support long-lived WebSocket connections. Route Handlers on the Edge Runtime handle the streaming cleanly within the 10-second function limit.
Dev phases — Six phases, each designed to be one or two focused Copilot sessions. The key rule: never ask the agent to do more than one phase at a time. Phase 1 builds the skeleton, Phase 2 builds auth, Phase 3 builds the UI shell — each is independently testable before you move on.
Constraints & tips — The Vercel and Firebase free tier limits are documented with concrete mitigations (ISR caching, Firestore batching, Edge Runtime). The Copilot tips section covers how to get the best out of your IDE setup for this specific project.
Pillars matrix — A one-page mapping of every engineering principle to the exact pattern and tool that satisfies it in this codebase.

Key architectural decisions
SSE over WebSockets — Vercel serverless does not support long-lived WebSocket connections on the free tier. Server-Sent Events (one-directional, HTTP-based) work perfectly within the 10-second function timeout by using Vercel's streaming responses and Edge Runtime.
Firebase Admin on the server — Never expose service account credentials to the client. All Firestore writes from webhooks go through the Admin SDK in Route Handlers.
Encrypted token storage — GitHub OAuth access tokens are AES-256 encrypted before being written to Firestore, using a TOKEN_ENCRYPTION_KEY env var.

Concurrency Security Performance
1
GitHub REST API client (server-side)
Octokit with the stored user token. Fetch repos, recent commits, open PRs. Cache with React Query staleTime.
2
Webhook ingestion route
/api/webhooks/github — verify X-Hub-Signature-256 HMAC, parse with Zod, write to Firestore.
3
SSE streaming route
/api/stream — uses Next.js Edge Runtime streaming. Firestore onSnapshot listener pushes events to the client.
4
EventSource client hook
useEventSource custom hook. Auto-reconnect with exponential backoff. Updates Zustand store on message.
5
Firestore security rules
Users can only read/write their own documents. Webhook writes go through Admin SDK (bypasses rules).
6
Rate limit GitHub API calls
Track remaining rate limit. Cache aggressively. Show rate-limit warning widget when near ceiling.

GitHub Copilot tips for this project
💬
Prompt Copilot with contextKeep types in scope. When writing a webhook handler, open the Zod schema and the Firestore model in split view — Copilot uses open tabs as context.
🧪
Generate tests alongside implementationAfter implementing a function, type // test: and describe what to test. Copilot generates Vitest specs that match your actual code.
🔐
Security-aware completionsAdd a comment like // validate HMAC before processing before a webhook handler. Copilot will include the signature check in the completion.
📐
Use Copilot Chat for refactoringHighlight a component and ask "refactor this to use Zustand instead of prop drilling" or "add proper error boundary wrapping".
📝
JSDoc comments = better completionsWrite JSDoc on your custom hooks and utility functions. Copilot's suggestions elsewhere in the project become significantly more accurate.
🗂
Keep a COPILOT_CONTEXT.mdOne file at the root describing the project, stack decisions, and conventions. Reference it in Copilot Chat at the start of complex sessions.
